# T+1 Settlement Compliance

## The Constraint
Under new SEC rules, cash equities settle at T+1. For a $100 cash account, you cannot day-trade endlessly; you can only trade with settled funds.

## Professional Implementation
- **Ledger Abstraction**: The CrewAI system does not query Alpaca's generic "buying power." It maintains a strict internal SQLite ledger tracking the exact minute a trade closes and unlocking those specific funds precisely 24 business hours later.

## Evolutionary Mechanism
If the system misses a trade because it incorrectly calculated the T+1 window over a holiday weekend, the Meta-Review Crew updates the ledger calculation logic to include a live API pull of the NYSE holiday calendar.

---

## 🛑 VC Challenge & Investor Critique
**The Clock Drift & Reconciliation Failure:** VCs who know market plumbing understand that T+1 isn't exactly 24 hours; it settles at the NSCC processing window (typically midnight of T+1). If your custom SQLite ledger unlocks funds at 14:00 PM because the trade closed at 14:00 PM yesterday, but the clearinghouse hasn't batch-processed it, your agent will attempt an illegal trade and the broker will reject it, potentially flagging the account.

### Critical User Journeys (CUJs)
1. **The Phantom Buying Power:** Agent sells AAPL on Monday at 10 AM. The internal SQLite ledger unlocks the funds on Tuesday at 10 AM. The agent buys TSLA at 10:05 AM. However, Alpaca's clearing firm hasn't released the batch settlement. *Result: Order rejected, execution flow crashes.*
2. **The LLM Hallucination:** The agent's prompt says "You have $10 settled." The agent decides it can use $10 for AAPL and another $10 for MSFT because it's bad at stateful math. *Result: Over-allocation error.*

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Source of Truth Enforcement**: Never rely entirely on the internal SQLite database for final execution authorization. The architecture must utilize a `PreTradeSettlementFlow`. Before order submission, a deterministic Python script must poll the Broker's explicit `settled_cash` API endpoint. The internal database is only used for *planning* (so the LLM knows what to research).
2. **Pydantic State Validation**: The LLM output must be parsed into a Pydantic model. We add a validator: `@validator('trade_amount') def check_funds(cls, v, values): if v > GlobalState.settled_cash: raise ValueError("Insufficient settled funds")`. This forces the CrewAI flow to automatically reject and re-prompt the agent if it hallucinates math.
