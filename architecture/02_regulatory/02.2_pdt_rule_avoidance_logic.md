# PDT Rule Avoidance Logic

## The Constraint
Pattern Day Trader (PDT) rules prohibit more than 3 day-trades in a rolling 5-business-day period for accounts under $25,000. However, this applies to Margin accounts, not Cash accounts.

## Professional Implementation
- **Account Type Enforcement**: The system is strictly initialized as a Cash Account via the Alpaca API to permanently bypass the PDT rule, shifting the compliance burden entirely to Good Faith Violations (GFVs).
- **Fallback Logic**: If the API detects a Margin configuration, the initialization script throws a fatal error and refuses to boot the CrewAI orchestrator.

## Evolutionary Mechanism
The Meta-Review Crew periodically audits the account configuration API. If it detects a broker policy change regarding micro-margin accounts, it flags the operator for a HITL (Human-in-the-loop) review.

---

## 🛑 VC Challenge & Investor Critique
**The Edge Case Blindspot:** While Cash Accounts bypass PDT, many retail brokers accidentally flag accounts or transition them if specific options trading permissions are enabled. If the system assumes PDT is impossible but the broker flags it anyway, the agent will continue attempting day-trades and get the account locked for 90 days.

### Critical User Journeys (CUJs)
1. **The Broker Override:** Alpaca's risk department flags the account and restricts it from opening new positions due to anomalous high-frequency requests, effectively simulating a PDT lock. The CrewAI system continues to generate "BUY" tasks, resulting in 500 API errors and wasting tokens.
2. **The Accidental Day Trade:** Even in a cash account, if the system buys and sells the same ticker on the same day with unsettled funds, it triggers a GFV. The agent confuses PDT logic with GFV logic.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Status Polling Flow**: Add a `BrokerHealthCheckNode` at the beginning of the daily CrewAI Flow. This deterministic node queries Alpaca's `account.status` and `account.daytrade_count` endpoints. If `account.status == 'ACCOUNT_RESTRICTED'`, the Flow enters an immediate hibernation state and pages the developer.
2. **Separation of Logic**: The LLM should not know or care about PDT or GFV rules. Injecting complex regulatory logic into the prompt wastes tokens and causes hallucination. Instead, abstract this into an `ExecutionGuardrailService` that sits between the Manager Agent and the Broker API.
