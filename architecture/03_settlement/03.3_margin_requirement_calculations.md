# Margin Requirement Calculations

## The Zero-Margin Enforcement
The $100 architecture is strictly a cash account. However, to prepare for Phase 3 (Live Deployment with scaling), the system must understand margin maintenance mathematics natively to avoid instant liquidations if the account ever transitions.

## Professional Implementation
- **Margin Simulation Tool**: A custom Python tool that calculates Initial Margin and Maintenance Margin requirements for any proposed ticker, regardless of whether the account is currently utilizing leverage.
- **Agent Awareness**: The Risk Manager Agent utilizes this tool to calculate the "Liquidation Risk" of a portfolio, ensuring the system never accidentally scales into heavily margined, highly volatile assets.

## Evolutionary Mechanism
The Meta-Review crew monitors the hypothetical margin requirements. If it notices that the chosen asset classes routinely trigger simulated margin calls during overnight gaps, it updates the Fundamental Agent's asset-screener to filter out high-maintenance-requirement securities.

---

## 🛑 VC Challenge & Investor Critique
**The YAGNI (You Aren't Gonna Need It) Trap:** VCs despise over-engineering for futures that don't exist yet. The account has $100 and is explicitly a Cash Account to avoid PDT rules. Wasting compute cycles, LLM context window, and agent debate time calculating hypothetical margin calls for a non-margin account is detrimental to current PnL. Every token spent analyzing simulated margin is a token not spent analyzing actual alpha.

### Critical User Journeys (CUJs)
1. **The Phantom Margin Call Panic:** The Risk Manager Agent gets confused between its simulated margin calculations and its actual cash account status. It panics over a "simulated margin call" and dumps a highly profitable position. *Result: Realized losses due to simulated risk.*
2. **Token Bloat:** The Margin Simulation Tool adds 400 tokens of context to every prompt for every trade. *Result: Unnecessary compute drag.*

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Strict Context Pruning**: We must ruthless prune the LLM's context. The `Margin Simulation Tool` is completely stripped from Phase 1 and Phase 2 architectures. The Risk Manager is explicitly prompted: "You operate a Cash Account. Margin does not exist. Focus solely on cash drawdown."
2. **Modular Architecture for Phase 3**: Margin logic is isolated into a separate Git branch/module (`margin_execution_flow.py`). It is only merged and activated by the CI/CD pipeline if the global configuration flag `ACCOUNT_TYPE` is explicitly switched to `MARGIN`. The CrewAI system remains lean and hyper-focused on current realities.
