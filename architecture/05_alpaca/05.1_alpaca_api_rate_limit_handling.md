# Alpaca API Rate Limit Handling

## The Compute Constraint
Alpaca imposes strict rate limits (e.g., 200 requests per minute). A multi-agent system debating in a loop can easily blast past this limit if it queries price data repeatedly.

## Professional Implementation
- **API Wrapper**: All CrewAI custom tools that interact with Alpaca are routed through a central `AlpacaRateLimiter` singleton class.
- **Backoff Logic**: If a 429 Too Many Requests response is caught, the wrapper implements an exponential backoff (`time.sleep(2**attempt)`), preventing the agents from crashing.

## Evolutionary Mechanism
The Meta-Review Crew analyzes logs for 429 errors. If it sees the Technical Agent caused 50 errors by polling the 1-minute candle tool excessively, it autonomously edits the agent's prompt: "CRITICAL: You are only allowed to call the price tool ONCE per evaluation."

---

## 🛑 VC Challenge & Investor Critique
**The Thread-Blocking Bottleneck:** A system architect will point out that `time.sleep()` inside an agent's tool execution blocks the entire execution thread. If the Crew is running asynchronously and one agent gets rate-limited and sleeps for 8 seconds, the entire orchestration pipeline stalls. During high volatility, an 8-second delay is an eternity.

### Critical User Journeys (CUJs)
1. **The Sleep Cascade:** Agent A hits the rate limit and sleeps. Agent B is waiting for Agent A's output. The market crashes. The Kill Switch is blocked because the main event loop is congested by sleeping threads. *Result: System lockup during critical market events.*
2. **The LLM Timeout:** The Alpaca tool sleeps for 15 seconds. The LangChain/CrewAI wrapper times out waiting for the tool to return, crashing the Crew task and losing the entire LLM context.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Centralized Data Caching**: The agents should almost never hit the Alpaca REST API directly for price data. The architecture must implement a local **Redis Cache**. A background async worker maintains the WebSocket stream and updates Redis. The agent's `PriceQueryTool` simply reads from local Redis (0ms latency, 0 rate limit impact).
2. **Circuit Breakers over Sleep**: For execution (which requires REST), if a 429 is hit, the tool should *not* sleep the thread. It should raise a custom `RateLimitException`. The CrewAI Flow catches this exception, aborts the current execution path gracefully, and queues it for a retry in a separate async worker, keeping the main loop unblocked.
