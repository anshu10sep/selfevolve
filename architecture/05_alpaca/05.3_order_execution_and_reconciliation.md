# Order Execution and Reconciliation

## The Final Mile
Generating a good trade idea means nothing if the execution logic fails. The system must confirm that orders actually fill.

## Professional Implementation
- **Order Submission**: The Judge Agent uses the `SubmitOrderTool` to post the trade to Alpaca.
- **Reconciliation Loop**: Immediately after submission, a background task polls the `GetOrderStatus` endpoint. If the order is `pending` for more than 5 minutes, it is autonomously canceled to free up tied capital.

## Evolutionary Mechanism
If the system detects a high rate of `canceled` orders due to tight limit prices failing to fill, the Meta-Review Crew steps in. It rewrites the Manager Agent's limit-price calculation prompt, instructing it to place limit orders 0.2% above the current ask to guarantee execution during momentum plays.

## 🛑 VC Challenge & Investor Critique
**The Hanging Order Vulnerability:** VCs evaluating trading infrastructure look for "hanging state" vulnerabilities. If you rely on a polling loop to check order status, and that loop crashes or the CrewAI task times out, the system loses track of the order. The order might fill hours later, completely unaccounted for in your internal ledger, destroying your risk management and triggering GFVs.

### Critical User Journeys (CUJs)
1. **The Blind Fill:** Agent submits a limit order. The CrewAI process crashes. The system restarts, assumes no orders are active, and buys something else. The original limit order fills. *Result: Account is massively over-leveraged and disjointed from reality.*
2. **The Canceled-But-Filled Race Condition:** The 5-minute timer expires. The system sends a `Cancel` request. In the 50 milliseconds it takes to reach Alpaca, the order fills. The system assumes it was canceled and updates the ledger to show free cash. *Result: Phantom buying power leads to clearinghouse rejection.*

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Webhook-Driven Reconciliation**: Do not rely on polling for state reconciliation. The architecture must implement a fast FastAPI webhook listener. Alpaca pushes order status updates (`fill`, `canceled`, `rejected`) directly to this webhook.
2. **Atomic Ledger Updates**: The webhook listener is the *only* component allowed to update the SQLite ledger. The CrewAI flow submits the order, records the `client_order_id`, and immediately ends. The system state is updated deterministically by the webhook via database transactions, eliminating hanging state vulnerabilities entirely.
