# CrewAI Process Types: Sequential vs Hierarchical

## Architectural Decisions
CrewAI supports two primary orchestration paradigms: Sequential and Hierarchical. For high-stakes financial trading, a hybrid approach is required.

## Professional Implementation
- **Hierarchical for Execution**: The core trading loop runs hierarchically. A `Portfolio Manager Agent` (acting as the Langchain-backed manager) sits at the top, delegating research and risk analysis dynamically to sub-agents based on the active state.
- **Sequential for Compliance**: Within the sub-crews, specific compliance routines (like the PDT and GFV checks) are forced into a strict Sequential process. This guarantees that an auditor agent always runs *after* a trade proposal but *before* the execution tool is invoked.

## Evolutionary Mechanism
The `Manager Agent` monitors the latency and token cost of both processes. If it detects that hierarchical delegation is causing excessive API calls due to agent "chatter," it self-edits its configuration to force more rigid Sequential routing for specific asset classes that require faster execution.

## 🛑 VC Challenge & Investor Critique
**The Hierarchical Latency Tax:** VCs focusing on trading infrastructure will argue that Hierarchical CrewAI introduces unacceptable latency. In a Hierarchical setup, the Manager LLM must read the prompt, decide which agent to call, wait for the agent, read the agent's output, and then decide the next step. This back-and-forth LLM routing can take 15-30 seconds per loop. By the time the Manager actually executes, the market has moved.

### Critical User Journeys (CUJs)
1. **The Delegation Death Loop:** The Manager asks the Technical Agent for RSI. The Technical Agent returns RSI. The Manager then asks the Technical Agent for MACD instead of asking for both at once. *Result: 45 seconds of LLM processing latency, missing the entry window.*
2. **The Bottlenecked Manager:** During a high-volatility event, 5 sub-crews send alerts simultaneously. The Manager Agent (a single LLM context window) gets overwhelmed, hits a token limit, and drops the risk-management alert.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Flow-Based Routing**: Hierarchical LLM routing is deprecated for execution. We must use **CrewAI Flows**. The routing logic is moved from an LLM Manager to a deterministic Python `RouterNode`. The Python code checks the regime (e.g., `if volatility > 20: route_to_flash_crew()`). This drops routing latency from 5 seconds to 1 millisecond.
2. **Sequential Only for Sub-Crews**: Agents operate *only* sequentially within specialized sub-crews. The "Manager" is just a deterministic accumulator node that synthesizes the Pydantic outputs of the sequential sub-crews.
