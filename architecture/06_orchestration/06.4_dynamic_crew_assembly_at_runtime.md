# Dynamic Crew Assembly at Runtime

## Adapting to Market Regimes
A rigid list of agents is inefficient. A high-volatility market requires different analysis than a low-volatility sideways market.

## Professional Implementation
- **The Factory Pattern**: The `Orchestrator` runs a lightweight pre-flight Crew whose only job is to evaluate the VIX and macroeconomic calendar for the day.
- **Dynamic Assembly**: Based on the pre-flight report, the system instantiates the specific agents needed. If volatility is extreme, it includes the `FlashCrashAuditorAgent`. If it's earnings season, it instantiates 5 parallel `FundamentalAnalystAgents` to handle the load.

## Evolutionary Mechanism
The pre-flight crew compares its historical dynamic assemblies against resulting PnL. If it notices that assembling the `Sentiment Analyst` during sideways markets yields no edge but costs API tokens, it evolves its factory logic to skip instantiating that agent under those specific regimes, optimizing computational cost autonomously.

**Cold Boot Latency:** Dynamic assembly sounds great in theory, but instantiating CrewAI agents, loading their long-term memory from Mem0, and initializing their LLM connections takes time. If you dynamically assemble a crew *after* a volatility spike is detected, the 3-5 seconds of cold-boot time guarantees you will be the last participant to enter the trade.

### Critical User Journeys (CUJs)
1. **The News Spike Miss:** CPI data drops at 8:30 AM. Volatility spikes. The Orchestrator spends 4 seconds dynamically assembling the `FlashCrashAuditorAgent` and spinning up tools. *Result: The massive first-minute candle is completely missed due to instantiation latency.*
2. **Memory Hydration Failure:** A dynamically assembled Fundamental Agent is spun up, but the vector database connection times out during initialization. *Result: The agent runs blind without historical context, hallucinating the company's baseline metrics.*

## đź› ď¸Ź Refinement & Architecture Update
To make this robust:
1. **Object Pooling (Warm Standby)**: We abandon "on-the-fly" instantiation during trading hours. Instead, we use an **Agent Object Pool**. All possible agents are instantiated and hydrated with memory at 08:00 AM (Pre-market). They are kept in RAM in a "Sleep" state. Dynamic assembly simply routes the CrewAI Flow to the pre-warmed agent in memory, dropping latency from 4 seconds to 1 millisecond.
2. **Event-Driven Flow Architecture**: The `Pre-Flight Crew` is replaced by a continuous deterministic `RegimeTrackerService` that updates a global state variable. The CrewAI Flow checks this variable at runtime: `if State.regime == 'EARNINGS_SEASON': execute_parallel_fundamental_flow()`.
