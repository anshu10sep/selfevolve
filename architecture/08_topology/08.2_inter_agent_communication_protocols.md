# Inter-Agent Communication Protocols

## Standardizing the Conversation
Unstructured conversation between agents leads to token bloat and hallucination. 

## Professional Implementation
- **Protocol Enforcement**: Agents are explicitly instructed in their backstories to communicate *only* via predefined Markdown templates or Pydantic JSON schemas.
- **Example**: The Fundamental Agent is forbidden from writing prose. It must output its findings using: `{"score": 8, "debt_ratio": 0.4, "catalyst": "Earnings beat"}`.

## Evolutionary Mechanism
If the Manager Agent detects that a sub-agent is generating overly verbose prose that wastes tokens, it triggers a `CorrectionCallback`. This callback dynamically rewrites the offending agent's prompt for the next task execution, explicitly enforcing strict JSON-only output constraints to evolve the communication efficiency.

---

## 🛑 VC Challenge & Investor Critique
**The Context Loss Critique:** The irony of standardizing everything to strict JSON is that you destroy the very reason you are using LLMs. If the Fundamental Agent is forced to reduce a massive, complex earnings call into `{"catalyst": "Earnings beat"}`, you have stripped away all the qualitative nuance (e.g., *how* the CEO spoke about forward guidance). A VC will point out that you are just building an expensive, slow, non-deterministic version of a traditional quant screener.

### Critical User Journeys (CUJs)
1. **The Nuance Erasure:** A CEO gives a terrible, unconfident earnings call, but technically beats EPS. The Fundamental Agent is forced into a JSON schema, outputs `{"score": 8, "catalyst": "EPS beat"}`, entirely dropping the qualitative red flags. *Result: The system buys into a sinking ship.*
2. **Schema Breakage:** The API provider updates their model weights. The LLM decides to add a helpful comment to the JSON `{"score": 8, "note": "Beware debt!"}`. The strict Pydantic parser crashes.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Dual-Channel Communication**: Sub-agents must output two distinct payloads per task:
   - `quantitative_payload`: Strict Pydantic JSON for the deterministic execution layer to calculate sizing and limits.
   - `qualitative_rationale`: A 100-word maximum string where the LLM is *required* to explain edge cases, risks, or nuances. 
2. **Flow-Based routing**: The CrewAI Flow passes the `quantitative_payload` to the math nodes, and only passes the `qualitative_rationale` to the final `Judge Agent` LLM. This preserves semantic intelligence without polluting the deterministic math.
