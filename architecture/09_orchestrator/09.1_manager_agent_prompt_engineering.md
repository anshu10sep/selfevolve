# Manager Agent Prompt Engineering

## The Core Identity
The Manager Agent must behave like a relentless, risk-averse institutional quant. Its prompt is the most critical file in the system.

## Professional Implementation
- **Backstory**: "You are a quantitative portfolio manager operating a highly constrained $100 fund. Your primary objective is capital preservation. You despise unnecessary risk. You view all sub-agent research with extreme skepticism."
- **Guardrails**: Explicit instructions detailing the rules of T+1 settlement and Good Faith Violations are embedded directly into the system prompt to ensure they are never abstracted away.

## Evolutionary Mechanism
The Manager Agent's prompt is technically a Jinja template linked to a database. As the system realizes profits or incurs losses, variables within the prompt—such as `current_risk_tolerance` or `maximum_drawdown_limit`—are automatically mutated by the Meta-Review Crew, continuously shifting the Manager's baseline personality between aggressive and defensive postures based on recent performance.

---

## 🛑 VC Challenge & Investor Critique
**The Semantic Saturation Problem:** System prompts with too many explicit rules ("Do not trigger GFV," "Max drawdown is 5%," "Hate risk") suffer from "attention dilution." The more rules you add to a prompt, the less likely the LLM is to follow any specific one perfectly. A VC will note that relying on a text string to enforce SEC settlement laws is regulatory suicide.

### Critical User Journeys (CUJs)
1. **The Attention Override:** The sub-agent proposes an incredibly lucrative, "once-in-a-decade" setup. The Manager Agent gets "excited" by the semantic tone of the sub-agent and completely ignores the "You despise risk" instruction in its backstory. *Result: Max drawdown limit breached.*
2. **The Evolved Cowardice:** The Meta-Review Crew lowers the `current_risk_tolerance` variable in the prompt so far that the Manager Agent refuses to take any trades for 6 months.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Strip Regulatory Logic from Prompts**: Remove all mentions of T+1, GFV, and strict numerical drawdown limits from the LLM prompt. The LLM cannot do math reliably. These rules are migrated entirely to the `ExecutionGuardrailService` (pure Python).
2. **Behavioral Anchor Prompts**: The prompt should focus exclusively on qualitative behavior: "Synthesize the provided data. Output your true conviction score based purely on the evidence. Ignore portfolio sizing." The Python wrapper around the LLM handles the actual risk application. This keeps the prompt lean and focused, maximizing the LLM's attention on analysis, not compliance.
