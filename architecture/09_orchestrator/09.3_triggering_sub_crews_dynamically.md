# Triggering Sub-Crews Dynamically

## Asynchronous Market Scanning
The Orchestrator must not waste resources constantly scanning the market when there are no setups.

## Professional Implementation
- **Event-Driven Architecture**: Integrating webhooks or WebSocket listeners that trigger the CrewAI execution pipeline only when specific market conditions are met (e.g., an RSI crossover on a watchlist asset).
- **Sub-Crew Activation**: Once triggered, the Orchestrator instantly spins up the deeply specialized Sub-Crews to analyze the specific anomaly.

## Evolutionary Mechanism
The Orchestrator evolves its own trigger conditions. If a specific RSI crossover trigger results in 10 consecutive false positives (failed trades), the system autonomously edits the triggering threshold script (e.g., changing the trigger from RSI < 30 to RSI < 25), evolving the entry filter before it even wakes up the CrewAI network.

---

## 🛑 VC Challenge & Investor Critique
**The Trigger Parameter Overfit:** Allowing the system to autonomously edit its own Python triggering scripts (e.g., changing RSI < 30 to RSI < 25) based on 10 consecutive failures is textbook curve-fitting. The AI is overfitting its parameters to the immediate past noise, ensuring it will fail when the regime shifts back. VCs will view this as mathematical suicide.

### Critical User Journeys (CUJs)
1. **The Over-Optimized Silence:** The AI keeps missing trades, so it keeps lowering the RSI trigger. It evolves the script to RSI < 15. The market enters a massive bull run where RSI never drops below 40. *Result: The system never triggers, missing months of gains.*
2. **The Sandbox Escape:** An LLM writing code that edits the production trigger script introduces a syntax error (`if RSI < 25` without a colon). *Result: The Python daemon crashes, system offline.*

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Banish Autonomous Code Editing**: The AI is absolutely forbidden from writing or editing the Python execution/trigger scripts. Code is immutable.
2. **Parameter Store Configuration**: The triggering daemon reads from a secure PostgreSQL database table `Trigger_Params`. The Meta-Review Crew is only allowed to update the *values* in this table via a strict Pydantic tool.
3. **Statistical Validity Checks**: Before updating the database from `RSI=30` to `RSI=25`, the Meta-Review Crew must run a `HistoricalBacktestTool` over a rolling 6-month window to prove the new parameter has positive Expected Value across *multiple* regimes, not just the last 10 trades.
