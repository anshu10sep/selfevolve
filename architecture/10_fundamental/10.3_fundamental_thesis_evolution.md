# Fundamental Thesis Evolution

## Long-Term vs Short-Term Memory
Fundamental metrics change quarterly, but price action changes minutely. The system must know how to weight its fundamental thesis over time.

## Professional Implementation
- **Knowledge Base**: The agent's fundamental thesis for an asset (e.g., "$AAPL is structurally bullish") is stored in a permanent vector database using Mem0.
- **Context Injection**: When the Technical Agent wants to trade $AAPL, the historical Fundamental thesis is instantly retrieved and injected into the Manager's context.

## Evolutionary Mechanism
The Fundamental Agent periodically reviews its own historical thesis against current reality. If it predicted strong revenue growth but the subsequent quarterly earnings missed expectations, the agent autonomously drafts a "Failure Reflection," updates its vector database, and permanently downgrades its internal conviction score for that specific asset class, evolving its macro-level bias.

---

## 🛑 VC Challenge & Investor Critique
**The Stale Vector Trap:** A VC will point out that Vector Databases (like Mem0/ChromaDB) retrieve information based on semantic similarity, not necessarily chronological truth. If the system stores "$AAPL is bullish" in January and "$AAPL is bearish" in April, a query in May for "AAPL thesis" might return the bullish January thesis simply because the keyword vector match was stronger. The Technical Agent gets fed dangerously outdated fundamental context.

### Critical User Journeys (CUJs)
1. **The Zombie Thesis:** A company goes bankrupt. The database contains 5 years of "Strong Buy" theses and 1 "Bankrupt" thesis. Semantic search pulls the 5 "Strong Buy" vectors. *Result: Agent tries to buy a delisted stock.*
2. **The Overlapping Memory Loop:** The agent drafts a "Failure Reflection" and stores it. The next time it trades, it pulls the reflection, gets depressed by its past failure, and drafts a *second* reflection about how bad it is. The vector DB fills with meta-angst instead of financial data.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Time-Decayed Vector Retrieval**: The RAG implementation must use a custom retriever that strictly weights by `timestamp`. Older theses are exponentially decayed. The prompt injected into the Manager must always be the *latest* chronologically validated thesis, completely bypassing pure semantic similarity matching for definitive financial outlooks.
2. **Key-Value Overwrite for State**: "Conviction Scores" should not be stored as text embeddings in a vector database. They are numerical state variables. They must be stored in a standard SQL/Redis key-value pair (`AAPL_Fundamental_Score: -0.5`). The vector DB is reserved *only* for the qualitative narrative rationale, completely separating quantitative state from semantic search.
