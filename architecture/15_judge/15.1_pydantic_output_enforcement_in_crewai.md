# Pydantic Output Enforcement in CrewAI

## Preventing Execution Hallucinations
The Judge Agent is the final gateway to the Alpaca API. If it outputs "I think we should buy 2 shares" instead of a machine-readable JSON, the API integration crashes.

## Professional Implementation
- **CrewAI Task Output**: Utilizing CrewAI's `output_pydantic` parameter on the final execution task.
- **Schema**: 
```python
class ExecutionSignal(BaseModel):
    ticker: str
    action: Literal['BUY', 'SELL', 'PASS']
    fractional_notional_value: float
    stop_loss_price: float
    take_profit_price: float
```
- **Execution**: CrewAI natively forces the LLM to loop until its output perfectly matches this schema.

## Evolutionary Mechanism
If the LLM loops excessively trying to format the schema, wasting API tokens, the `Meta-Review Crew` intercepts this telemetry. It automatically updates the Judge Agent's prompt to include three perfect "few-shot" examples of the exact Pydantic JSON structure, permanently evolving its formatting efficiency.

## 🛑 VC Challenge & Investor Critique
**The Syntax Retry Token Burn:** As highlighted previously, relying on CrewAI's native Pydantic retry loop is an amateur mistake in production. If the LLM generates a string for a float field (`"stop_loss": "fifteen dollars"`), CrewAI throws an error back to the LLM. The LLM might retry 3 times, consuming 15,000 context tokens. At $5.00 per 1M input tokens (GPT-4o), a single bad JSON string could cost you $0.07 in retries, instantly destroying the margin on a micro-capital trade.

### Critical User Journeys (CUJs)
1. **The Negative Stop-Loss:** The agent correctly outputs JSON, but hallucinates a mathematical impossibility (`stop_loss_price: -10.50`). Pydantic's basic type checking accepts it because it is a valid float. Alpaca rejects it. Execution fails.
2. **The "None" Bypass:** The LLM isn't confident in a take-profit target, so it outputs `take_profit_price: None`. The Pydantic model (expecting a float) crashes the Crew task.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Rigorous Pydantic Validators**: Basic typing is insufficient. The `ExecutionSignal` model must utilize advanced `@field_validator` methods.
```python
@field_validator('stop_loss_price')
def validate_positive(cls, v):
    if v <= 0: raise ValueError("Price must be > 0")
    return round(v, 2)
```
2. **Provider-Native Structured Outputs**: The retry loop must be entirely circumvented by leveraging `instructor` or the native Anthropic/OpenAI "Tool Use" / "Structured Outputs" API parameters. This guarantees the LLM's raw byte output strictly adheres to the schema on the *first* attempt, eliminating retry token burn.
