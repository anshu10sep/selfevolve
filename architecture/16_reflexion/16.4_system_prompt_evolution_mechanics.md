# System Prompt Evolution Mechanics

## Self-Modifying Code Constraints
Modifying prompts dynamically is powerful but dangerous. An agent could accidentally erase its core instructions.

## Professional Implementation
- **Prompt Architecture**: The system prompt is divided into `Immutable_Core` (e.g., "Do not trigger GFV violations") and `Mutable_Strategy` (e.g., "Favor tech stocks").
- **Sandboxed Evolution**: The Meta-Review Crew is only granted permissions via its custom tool to edit the `Mutable_Strategy` block.

## Evolutionary Mechanism
To ensure the evolution doesn't degrade performance, the system maintains a Git-like version control of the `Mutable_Strategy`. If the newly evolved prompt results in 3 consecutive losses the next day, the `Portfolio Manager Agent` detects the degradation and automatically rolls back the `Mutable_Strategy` string in the database to the previous `v1.2` version, preventing catastrophic evolutionary drift.

---

## 🛑 VC Challenge & Investor Critique
**The Version Control Illusion:** Rolling back to `v1.2` after 3 consecutive losses assumes that `v1.2` was actually a good prompt and not just lucky. A VC will challenge that 3 trades is a statistically insignificant sample size to determine if a prompt is "degraded." You are reacting to noise, not signal. Constantly rolling back prompts creates a thrashing effect where the AI never actually learns.

### Critical User Journeys (CUJs)
1. **The Noise Thrash:** Prompt `v1.2` is generated. It loses 3 times due to random market noise (e.g., a sudden tweet by a politician). The system rolls back to `v1.1`. `v1.1` was actually a terrible prompt for the current regime, and the system loses 10 more times.
2. **The Database Bloat:** The system updates the prompt every night, creating thousands of Git-like versions. The tracking script becomes too slow to query the optimal rollback version.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Statistical Significance Testing**: A rollback cannot be triggered by "3 consecutive losses." We must use a deterministic **Z-Score or T-Test calculation** in Python. The system compares the win rate of `v1.2` against `v1.1` over a minimum of 20 live/paper executions. If the difference is statistically significant (p < 0.05) in the negative direction, *only then* does it rollback.
2. **Parallel Shadow Simulation**: As proposed earlier, never let a brand new, unproven prompt manage the live $100 account. The `Mutable_Strategy_v1.2` must be deployed to a **Shadow Crew** that runs on paper-trading concurrently with the live `v1.1` crew. If the Shadow Crew outperforms the Live Crew over 5 days, the database promotes `v1.2` to Live. This guarantees evolution is always verified mathematically before risking real capital.
