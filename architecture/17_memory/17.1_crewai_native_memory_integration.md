# CrewAI Native Memory Integration

## Short-Term vs Entity Memory
CrewAI offers native `memory=True` functionality, which enables short-term, long-term, and entity memory.

## Professional Implementation
- **Configuration**: `Crew(agents=[...], tasks=[...], memory=True)`
- **Under the Hood**: This automatically spins up local ChromaDB or SQLite instances to store the inter-agent chatter during a specific session, allowing the Manager to recall what the Fundamental Analyst said 3 tasks ago without re-prompting.

## Evolutionary Mechanism
The native memory allows the Crew to evolve its *intraday* strategy. If a stock was analyzed at 10:00 AM and rejected, and the ticker appears again at 14:00, the native Entity memory recognizes the ticker, instantly recalling the morning's rejection logic, saving compute time and evolving its efficiency continuously throughout the day.

---

## 🛑 VC Challenge & Investor Critique
**The Context Contamination Risk:** Native `memory=True` in basic agent frameworks is often a black box. If the system "remembers" that it rejected a stock at 10:00 AM, it might aggressively reject it at 14:00 PM out of bias, *even if the underlying facts changed completely* (e.g., an earnings report dropped at 13:00). Relying on black-box native memory introduces a massive cognitive anchoring bias into the AI.

### Critical User Journeys (CUJs)
1. **The Anchoring Bias Fatal Error:** 10:00 AM: Agent rejects AAPL because RSI is overbought. 14:00 PM: AAPL drops 5%, RSI resets to oversold, creating a perfect entry. The agent is queried again. Native memory injects "We rejected AAPL earlier." The agent ignores the new data and rejects the trade based on stale memory. *Result: Missed alpha due to memory bias.*
2. **The Memory Leak:** The local ChromaDB instance spun up by `memory=True` is not aggressively pruned. After 5 days of trading, the local SQLite database hits 5GB, causing severe I/O latency, crashing the Crew task on a slow disk read.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Disable Black-Box Memory**: We explicitly set `memory=False` in the CrewAI configuration. We cannot tolerate opaque, uncontrollable state injection in a financial application.
2. **Explicit State Passing (Flows)**: Intraday "memory" is simply state. We use CrewAI Flows to pass a strictly defined Pydantic `SessionState` object between nodes. If AAPL was analyzed at 10:00 AM, it is added to `SessionState.analyzed_tickers` with a `timestamp`.
3. **Time-Gated Recall**: When the Python Router Node sees AAPL again at 14:00, it checks the timestamp. `if (current_time - timestamp) > 60_minutes: force_re_analysis()`. This mathematically forces the LLM to look at fresh data without bias, completely solving the anchoring problem.
