# Mem0 Entity Extraction Tools

## Global Permanent Memory
CrewAI's native memory is often scoped to the run. For long-term self-evolution across weeks and months, an external provider like Mem0 is required.

## Professional Implementation
- **Tool**: `GlobalMemoryReadWriteTool`
- **Entity Extraction**: When the Meta-Review Crew generates a reflection, it extracts specific entities: `[Ticker: AAPL]`, `[Regime: High Volatility]`, `[Mistake: Bought pre-earnings]`.
- **Mem0 Push**: These entities are pushed to the Mem0 API as a permanent memory node.

## Evolutionary Mechanism
When the `Manager Agent` initializes a new trade proposal for AAPL a month later, it automatically queries Mem0 with the ticker. Mem0 returns the historical "Bought pre-earnings" failure. The Manager Agent immediately injects this memory into the Debate workflow, ensuring the system never repeats the exact same mistake twice, achieving true long-term evolution.

**The API Latency & Vendor Lock-in:** Relying on an external cloud API (like Mem0) for critical path execution introduces unacceptable latency and vendor lock-in. If the Mem0 API takes 800ms to return the entity graph for a ticker during a live trade, your execution window is compromised. Furthermore, sending proprietary trading logic and alpha-generating reflections to a third-party cloud provider is a massive compliance and intellectual property risk.

### Critical User Journeys (CUJs)
1. **The API Timeout:** The system attempts to query Mem0 for AAPL during a high-volatility event. Mem0 is experiencing an AWS outage. The `GlobalMemoryReadWriteTool` hangs for 10 seconds, crashes the CrewAI task, and misses the trade.
2. **The PII / Alpha Leak:** The Meta-Review Crew inadvertently includes the exact proprietary parameters of its breakout strategy inside the Mem0 payload. This data is now stored on a third-party server, violating internal IP policies.

To make this robust:
1. **Local Self-Hosted RAG**: We completely rip out the external Mem0 API dependency. We deploy a lightweight, self-hosted **Qdrant** or **Chroma** container directly alongside the CrewAI orchestration Docker image. This guarantees <10ms retrieval latency and 100% data sovereignty.
2. **Asynchronous Memory Writes**: The `GlobalMemoryReadWriteTool` must never block the execution thread. When the Meta-Review Crew generates a reflection, it pushes the JSON payload to a local Redis Queue. A separate, low-priority Python worker consumes the queue, handles the vector embedding (using a local model like `all-MiniLM-L6-v2`), and writes to Qdrant asynchronously.
