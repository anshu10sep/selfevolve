# CrewAI Ask Human Input Callbacks

## Bridging the Gap safely
While autonomous, high-risk trades must ping the operator. CrewAI supports `human_input=True` at the agent level, but we need granular control.

## Professional Implementation
- **Custom Tool**: Instead of blocking the whole process, we use a `HumanApprovalTool`.
- **Workflow**: If the Judge Agent triggers an interrupt, it calls the `HumanApprovalTool` via an async webhook (e.g., sending a Telegram message with `Approve/Reject` buttons).
- **Execution Pause**: The CrewAI task naturally waits for the tool to return a string response before proceeding.

## Evolutionary Mechanism
If the human operator clicks `Reject` and provides a reason via Telegram (e.g., "Too close to market close"), the `HumanApprovalTool` captures this text. The Meta-Review Crew reads this text overnight, adding "Do not trade within 30 minutes of market close" to the Judge's permanent constraints, evolving the system to align perfectly with the operator's unwritten preferences.

---

## 🛑 VC Challenge & Investor Critique
**The Synchronization Bottleneck:** Pausing an autonomous orchestration pipeline to wait for a human via an async webhook creates severe state management risks. If the CrewAI task is "waiting" for a string response, the underlying Python thread is hanging. If the human takes 4 hours to click "Approve" on Telegram, the trade setup has vanished, but the system will execute it the millisecond the button is clicked, resulting in a disastrous, late fill.

### Critical User Journeys (CUJs)
1. **The Stale Approval:** System proposes a breakout buy at 10:00 AM. Human is in a meeting. Human clicks "Approve" at 11:30 AM. The stock has already broken out and crashed back down. The system buys the absolute top because it was blindly waiting for the webhook.
2. **The Memory Leak:** 5 trades trigger the `HumanApprovalTool` simultaneously. The human ignores them. 5 CrewAI task threads hang indefinitely in memory, eventually causing an Out-Of-Memory (OOM) exception that crashes the entire container.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Time-to-Live (TTL) Hard Kills**: The `HumanApprovalTool` must have an aggressive, deterministic TTL. `timeout=60_seconds`. If the human does not click the Telegram button within exactly 60 seconds, the Python tool automatically returns `"System_Reject: Timeout"`. The system assumes rejection by default, clearing the thread and protecting the portfolio from stale data execution.
2. **State Serialization**: Never hang the thread. When the tool fires the Telegram webhook, the CrewAI Flow must serialize its exact state (ticker, notional value, limit price) to PostgreSQL and *terminate the process entirely*. When the human clicks "Approve," the Telegram webhook spins up a *brand new* lightweight Python execution script that pulls the state from the DB, verifies the price hasn't moved >0.1%, and executes, completely decoupling human latency from the LLM engine.
