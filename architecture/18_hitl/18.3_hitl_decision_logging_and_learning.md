# HITL Decision Logging and Learning

## Learning from the Master
When the human intervenes, it is the highest-fidelity training data the system can receive.

## Professional Implementation
- **Database Schema**: A dedicated SQL table `hitl_interventions` storing: `[Timestamp, Proposed_Trade, Human_Decision, Human_Comment, Market_Context]`.
- **Forced Ingestion**: The Meta-Review Crew is hardcoded to process any new rows in this table with priority over all other reflection tasks.

## Evolutionary Mechanism
This creates a behavioral cloning loop. If the human operator repeatedly rejects trades on Friday afternoons, the system extracts the `[Day_Of_Week: Friday]` and `[Time: PM]` features. It autonomously creates a new strict constraint in the `Manager Agent`'s prompt, effectively evolving to mimic the human's schedule preferences without explicitly being coded to avoid Fridays.

## 🛑 VC Challenge & Investor Critique
**The Spurious Correlation Fallacy:** VCs investing in ML models fear spurious correlations. If an LLM (the Meta-Review Crew) is tasked with finding patterns in human interventions, it is highly likely to hallucinate the wrong causal link. If you reject three trades on Friday because the *underlying assets had bad charts*, the LLM might incorrectly deduce that the *day of the week* is the problem, evolving a completely illogical rule that harms future PnL.

### Critical User Journeys (CUJs)
1. **The Hallucinated Rule:** A human rejects three AAPL trades because the CEO was testifying before Congress. The LLM sees the data, notices it was raining in New York on those three days, and evolves a rule: "Never trade AAPL when precipitation > 0." *Result: Nonsensical trading logic installed in production.*
2. **The Operator Error Cloned:** The human operator is tired and accidentally clicks "Reject" on a mathematically perfect trade setup. The system clones this mistake, evolving a permanent bias against a highly profitable setup.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Structured Human Intent**: The system cannot be left to guess *why* the human rejected the trade. The Telegram Webhook must enforce a dropdown menu of specific rejection reasons: `[Macro Risk, Technical Disagreement, News Event, Other]`. The `Human_Comment` is secondary.
2. **Deterministic Feature Extraction**: The Meta-Review Crew (LLM) is banned from autonomous feature extraction. We use a deterministic statistical algorithm (e.g., Random Forest feature importance) running on the `hitl_interventions` table over a minimum of 50 data points. If the algorithm proves that `Time: PM` is statistically the highest correlated feature to rejection, *only then* does it generate a prompt update for the Manager Agent.
