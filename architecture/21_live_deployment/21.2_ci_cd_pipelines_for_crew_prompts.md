# CI/CD Pipelines for Crew Prompts

## Treating Language as Code
If an agent's prompt changes, the behavior of the entire trading system changes. Prompts must be version-controlled like Python scripts.

## Professional Implementation
- **GitHub Actions**: A CI/CD pipeline triggered on every commit to the `prompts/` directory.
- **Validation**: The pipeline runs a suite of PyTest unit tests. It uses a mocked `gpt-4o-mini` instance to verify that the new prompt still outputs the required `ExecutionSignal` Pydantic schema before allowing the merge to `main`.

## Evolutionary Mechanism
When the `Meta-Review Crew` autonomously rewrites a prompt, it does not overwrite the live database directly. It programmatically opens a Pull Request (PR) on GitHub. The CI/CD pipeline automatically runs a 5-day historical backtest on the proposed prompt. If the backtest shows a higher Sharpe Ratio than `main`, the pipeline auto-merges the PR and triggers an ECS redeployment.

---

## 🛑 VC Challenge & Investor Critique
**The Backtest Illusion & CI/CD Lag:** As noted previously, running a "5-day historical backtest" using LLMs in a CI/CD pipeline is fundamentally flawed due to lookahead bias and the extreme time/cost required to run thousands of API calls during a GitHub Action. Furthermore, if a critical bug is discovered intraday, having to wait 20 minutes for a GitHub Action to build a Docker image and deploy it to ECS means your portfolio is bleeding cash while you wait for the pipeline to turn green.

### Critical User Journeys (CUJs)
1. **The Overfit Auto-Merge:** The LLM generates a prompt that overfits to the last 5 days of noise. The CI/CD backtest sees a high Sharpe ratio and auto-merges. The bad prompt is deployed to live trading and immediately loses money.
2. **The Deployment Bleed:** The live system enters an infinite error loop due to an Alpaca API change. The developer commits a 1-line Python fix. The GitHub Action takes 15 minutes to run its LLM prompt tests and build the Docker image. The $100 account is liquidated in minute 5.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Hot-Swapping over Docker Builds**: Prompts must NOT be baked into the Docker image. They must reside in a live, in-memory Redis cache or PostgreSQL database. When a prompt is updated, the ECS containers simply fetch the new string dynamically on the next task loop. This reduces deployment time from 15 minutes to 1 millisecond.
2. **Human-Gated Evolution PRs**: The `Meta-Review Crew` is allowed to open the GitHub PR with its proposed prompt. The CI/CD pipeline runs *syntax and schema validation only*. It does NOT run historical backtests. The PR requires a manual human `Approve` click to merge. Live financial evolution is too dangerous for 100% auto-merging based on synthetic metrics.
