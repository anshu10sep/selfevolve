# Alpaca SIP Data Ingestion Pipeline

## The Lifeblood of the System
A trading AI is only as intelligent as the data it consumes. Retail delayed data is insufficient.

## Professional Implementation
- **Subscription**: Utilizing Alpaca's SIP (Securities Information Processor) data feed for 100% consolidated, real-time tick data across all US exchanges.
- **WebSocket Daemon**: A dedicated Python asynchronous worker subscribes to `trades`, `quotes`, and `minute_bars` for the active watchlist.

## Evolutionary Mechanism
If the system detects that it is consistently losing trades due to slippage on specific high-beta assets, the Meta-Review Crew autonomously instructs the WebSocket Daemon to subscribe to the raw `orderbooks` (Level 2 data) for those specific tickers, allowing the Technical Agent to analyze bid/ask imbalances before execution.

---

## 🛑 VC Challenge & Investor Critique
**The Level 2 Data Tsunami:** VCs know that Level 2 order book data is an absolute firehose. A single highly liquid ticker (like SPY or AAPL) can generate thousands of order book updates per second. If an LLM (the Meta-Review Crew) carelessly tells the Python daemon to subscribe to Level 2 data for 10 tickers, the Python daemon will instantly be overwhelmed, Max out CPU/Memory, and crash. You cannot feed Level 2 data into an LLM agent architecture; the LLM takes seconds to read what changes in microseconds.

### Critical User Journeys (CUJs)
1. **The L2 Memory Out-Of-Memory (OOM) Crash:** The system subscribes to Level 2 data. 50,000 JSON payloads hit the daemon in 10 seconds. The daemon tries to store them in RAM. The Docker container hits its memory limit and OOMs. The system dies.
2. **The Pointless Context Injection:** The daemon successfully captures Level 2 data and dumps the top 5 bids/asks into the Technical Agent's prompt. By the time the LLM finishes reading the prompt 3 seconds later, the entire order book has shifted 40 times. The LLM's analysis is completely irrelevant.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Ban LLM Level 2 Analysis**: The LLM Intelligence Layer is strictly forbidden from accessing or analyzing Level 2 order book data. LLMs operate on semantic, minute-level timeframes.
2. **Deterministic Imbalance Scoring**: If Level 2 data is required, it must be handled by a dedicated, high-performance C++ or Cython service. This service calculates a simple integer: `Order_Book_Imbalance_Score` (-100 to +100). 
3. **Low-Frequency Injection**: Only this single integer (e.g., `+45`) is injected into the CrewAI Flow. The LLM understands "The buy-side is heavy," without ever having to parse the microscopic chaos of the actual order book.
