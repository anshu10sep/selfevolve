# PostgreSQL Time-Series Storage

## Localizing Data for Speed
Constantly querying the Alpaca historical API for backtesting or intraday analysis triggers rate limits and slows down the Crew.

## Professional Implementation
- **TimescaleDB**: Utilizing PostgreSQL with the TimescaleDB extension to optimize for fast, continuous inserts of time-series financial data.
- **Continuous Ingestion**: The WebSocket Daemon writes every incoming 1-minute bar directly into the TimescaleDB. The Technical Agent queries this local database instead of the external API.

## Evolutionary Mechanism
The database includes an automated cron job that down-samples old data. If the system's hard drive approaches 80% capacity, an autonomous database agent executes a script to compress 1-minute bars older than 30 days into 1-hour bars, preserving the long-term trend data while efficiently managing infrastructure costs.

---

## 🛑 VC Challenge & Investor Critique
**The I/O Bottleneck & Race Conditions:** Writing every 1-minute bar for a 500-stock watchlist into a relational database (PostgreSQL) in real-time introduces massive disk I/O bottlenecks. If the Technical Agent queries the database at exactly 10:01:00 to make a trade decision, but the WebSocket Daemon's INSERT query for the 10:01 bar is queued behind other database transactions, the Agent will read stale data and make a flawed decision.

### Critical User Journeys (CUJs)
1. **The Stale Read:** It's 10:05:02. The Technical Agent queries the DB for the 10:05 close. The DB is still processing the batch insert. The DB returns data up to 10:04. The agent executes a breakout trade based on old data, missing the 10:05 reversal.
2. **The Compression Corruption:** The autonomous database agent compresses 1-minute bars into 1-hour bars to save space. However, the system's backtesting script relies on 5-minute bars. The historical backtester crashes because the granularity it needs was deleted by the LLM's storage optimization.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Redis for Real-Time, Postgres for History**: PostgreSQL/TimescaleDB is too slow for the "live edge." We must implement a dual-layer architecture.
   - **Hot State (Redis)**: The WebSocket Daemon writes the current day's 1-minute bars directly to an In-Memory Redis Cache. The Technical Agent *only* reads from Redis for live trading, guaranteeing sub-millisecond retrieval of the absolute latest data.
   - **Cold State (Postgres)**: A background worker asynchronously flushes the Redis data to TimescaleDB at the end of the trading day for permanent historical storage.
2. **Immutable Granularity**: Storage is cheap; alpha is expensive. We ban the system from "down-sampling" 1-minute bars. If storage exceeds 80%, the system simply pages the developer to increase the AWS EBS volume size. We never delete high-fidelity historical data.
