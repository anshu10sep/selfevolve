# Automated Process Restarts

## The Inevitability of Crashes
Python scripts crash. APIs timeout. Docker containers run out of memory. The system must be self-healing.

## Professional Implementation
- **Supervisord / PM2**: The core CrewAI execution script is wrapped in a process manager (e.g., Supervisord inside the Docker container).
- **Auto-Restart**: If the script throws a fatal exception (e.g., `requests.exceptions.ConnectionError` from OpenAI), Supervisord instantly catches the exit code and restarts the Python process.

## Evolutionary Mechanism
The process manager logs every crash reason to a text file. The `Meta-Review Crew` reads this file on boot. If it sees "RateLimitError" caused the last 3 crashes, it autonomously edits the API configuration file to increase the global `time.sleep()` backoff parameter, evolving the system's network resilience to prevent future crashes.

## 🛑 VC Challenge & Investor Critique
**The Restart Death Loop:** "Auto-Restarting" a failed trading daemon is one of the most dangerous things you can do in production. If the Python script crashes because there is a bug in the Alpaca API wrapper that submits a market order incorrectly, Supervisord will restart the script. The script will boot up, see the trade setup, execute the buggy market order again, crash again, and restart again. It will execute 50 buggy market orders in 5 seconds before the exchange bans your account.

### Critical User Journeys (CUJs)
1. **The Poison Pill Trade:** The LLM generates a Pydantic schema that causes a database constraint error on insertion. The script crashes. Supervisord restarts it. CrewAI loads its state, attempts the exact same database insertion, and crashes again. *Result: System is locked in a boot-loop, unable to manage open positions.*
2. **The LLM Config Corruption:** The Meta-Review Crew autonomously edits the API configuration file to increase `time.sleep()`. It introduces a YAML syntax error. Supervisord restarts the script. The script can't parse the YAML and crashes immediately. *Result: Complete system bricking due to autonomous configuration editing.*

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Stateful Re-entry Barriers**: A process manager like Supervisord must be configured with strict `startretries=3`. If the script crashes 3 times in 60 seconds, it must enter a `FATAL` state and stay dead.
2. **Idempotent Execution**: The system must be perfectly idempotent. Upon boot, before doing *anything*, it must query the broker. "Do I have an order currently pending? Are my local DB states perfectly synced with Alpaca?" If there is any discrepancy, the system enters `RECONCILIATION_MODE`, refusing to generate new trades until the state is manually or mathematically resolved.
3. **Immutable Configs**: Reiterate the absolute rule: The LLM is banned from editing `.py`, `.yaml`, or `.json` configuration files. Evolution is strictly limited to prompts and database weights.
