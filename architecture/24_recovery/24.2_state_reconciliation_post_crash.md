# State Reconciliation Post-Crash

## Re-synchronizing with Reality
When a CrewAI process dies mid-task, its internal memory (what trades it thought it placed) is wiped. Upon reboot, it must rebuild its worldview.

## Professional Implementation
- **The Boot Sequence**: 
  1. Ping Alpaca for all open positions.
  2. Ping Alpaca for all pending orders.
  3. Query the internal SQLite ledger for available settled cash.
- **Context Hydration**: This live data is injected directly into the `Portfolio Manager Agent`'s prompt on boot: "System recovered from crash. Current portfolio: 1 AAPL share. Pending orders: None."

## Evolutionary Mechanism
If the system crashes while a limit order was pending, and upon reboot it realizes the order filled while it was offline, the `Meta-Review Crew` identifies this "blind spot." It autonomously updates the boot sequence script to automatically place a hard stop-loss on any newly discovered, unmanaged positions immediately upon boot, evolving its defensive posture post-crash.

---

## 🛑 VC Challenge & Investor Critique
**The Execution Overlap & Hallucinated State:** Relying on the LLM (`Portfolio Manager Agent`) to read the prompt ("System recovered... Current portfolio: 1 AAPL") and figure out what to do next is extremely fragile. The LLM might hallucinate that it was *supposed* to sell AAPL before the crash, and immediately issue a market sell order, disrupting the actual strategy that the previous instance was attempting to execute. The LLM does not have continuous consciousness.

### Critical User Journeys (CUJs)
1. **The Double Execution:** The system decides to buy AAPL, sends the API request, and crashes 1 millisecond later before recording it in the SQLite DB. Upon reboot, it sees AAPL in the Alpaca account. The LLM gets confused because its last memory was "I should buy AAPL." It buys a *second* share of AAPL, double-allocating the portfolio.
2. **The Stop-Loss Race Condition:** The LLM evolves the boot script to "automatically place a hard stop-loss." The system reboots, sees 1 share of TSLA, and the Python script fires a stop-loss order. However, the original process had *already* placed a stop-loss at the broker before crashing. Now there are two stop-loss orders for 1 share, which will cause the broker to reject one or both, or create a short position if triggered.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Client Order IDs (Idempotency Keys)**: Every single order sent to Alpaca must use a deterministic `client_order_id` (e.g., `buy_AAPL_20261026_1430`). If the system crashes and reboots, it queries Alpaca. If it decides to execute the trade again, Alpaca will reject the duplicate `client_order_id`, physically preventing double execution.
2. **Deterministic Reconciliation, Not LLM Prompting**: The LLM should *not* be told about the crash. The Python execution layer handles reconciliation entirely. If an unmanaged position is found without an attached bracket order (Stop Loss / Take Profit), a pure Python script attaches a default 5% ATR trailing stop immediately. The LLM is only awakened *after* the portfolio is mathematically secured.
