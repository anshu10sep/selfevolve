# Broker Position Syncing Scripts

## Preventing the Shadow Ledger
The system's internal ledger and the broker's actual database will eventually drift due to fractional rounding or rejected orders.

## Professional Implementation
- **Cron Job**: A deterministic Python script runs every day at 16:15 EST (Post-Market Close).
- **Action**: It pulls the exact `equity`, `cash`, and `positions` from Alpaca. It completely overwrites the local SQLite ledger, ensuring tomorrow's session starts with mathematically perfect data.

## Evolutionary Mechanism
If the sync script consistently finds a -$0.05 drift in the cash balance every day, the `Meta-Review Crew` investigates. It deduces that Alpaca's fractional routing incurs hidden micro-slippage not accounted for in the execution math. It autonomously updates the `Capital_Reservation_Tool` to reserve an extra 0.5% cash buffer on every trade, evolving the system to account for invisible broker fees.

---

## 🛑 VC Challenge & Investor Critique
**The End-of-Day Blindness:** Overwriting the internal ledger only at 16:15 EST guarantees that your system is operating on completely flawed math during the most critical hours of the trading day. If a -$0.05 drift occurs at 10:00 AM, and the agent tries to use its entire balance at 14:00 PM, the Alpaca API will reject the trade for insufficient funds. An autonomous system cannot wait until the market closes to realize its math is wrong.

### Critical User Journeys (CUJs)
1. **The Intraday Drift Rejection:** The LLM ledger says $10.00. Alpaca actually has $9.95 due to slippage on a morning trade. At 15:50 PM, the system sees a perfect setup and tries to buy $10.00. Alpaca rejects it. The setup is missed because the sync script doesn't run for another 25 minutes.
2. **The Destructive Overwrite:** A bug in the Alpaca API temporarily reports $0.00 equity at 16:15 EST. The sync script blindly overwrites the internal SQLite database with $0.00. The next morning, the AI thinks the account is bankrupt and shuts down.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Continuous Ledger Validation**: We abandon the end-of-day overwrite. The CrewAI Flow must validate its state **Just-In-Time (JIT)**. Before the `ExecutionNode` fires an order, it makes a lightweight `GET /account` request to Alpaca. It compares `Alpaca.cash` to `SQLite.cash`. If there is a discrepancy, the system syncs *in that exact millisecond*, adjusts the position size, and executes.
2. **Anomaly Alerting**: If the sync script detects a drift > 1% of total equity, it does *not* blindly overwrite the database. Massive drift indicates a structural failure in the execution logic. The system must immediately page the developer: `CRITICAL: Ledger Drift Detected. Halting trading.`
