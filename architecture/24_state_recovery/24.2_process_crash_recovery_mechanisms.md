# Process Crash Recovery Mechanisms

## Resuming the Crew
If the server loses power mid-debate, the CrewAI process dies. It must resume exactly where it left off.

## Professional Implementation
- **Task State Checkpointing**: CrewAI allows tasks to be executed sequentially. We wrap the `Crew.kickoff()` method in a checkpoint manager.
- **Database Logs**: As each `Task` completes, its output is serialized to disk. Upon restart, the `CrewFactory` reads the disk. If Task 1 and 2 are complete, it skips them, injecting their outputs directly into the context of Task 3, bypassing the LLM calls entirely.

## Evolutionary Mechanism
If the system crashes specifically when querying massive SEC filings because the server runs out of RAM, the crash logger catches the `OOMKill` signal. Upon recovery, the `Manager Agent` reads the crash log and permanently evolves the `SecEdgarTool` configuration to request smaller paginated chunks of data, preventing future Out-of-Memory crashes.
