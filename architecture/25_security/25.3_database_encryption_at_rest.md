# Database Encryption at Rest

## Protecting Proprietary Alpha
The central database contains the system's evolved prompts, backstories, and decision logic. This is the intellectual property of the fund.

## Professional Implementation
- **Encryption**: The PostgreSQL database backing the CrewAI memory and configuration is deployed on AWS RDS with AES-256 encryption at rest enabled.
- **Access Control**: The ECS tasks connect to the database using IAM database authentication, completely eliminating hardcoded passwords.

## Evolutionary Mechanism
The system monitors AWS CloudTrail logs. If it detects an unauthorized IP address attempting to query the PostgreSQL database, the Orchestrator autonomously triggers a `LockdownEvent`. It rotates the IAM roles, restricts security group access to the VPC exclusively, and pages the administrator, evolving its network security perimeter in real-time.

---

## 🛑 VC Challenge & Investor Critique
**The CloudTrail Latency & Scope Creep:** CloudTrail logs are not real-time; they can have a 15-minute ingestion delay. If a malicious actor breaches the VPC, they can dump the entire PostgreSQL database in 30 seconds. A 15-minute delayed `LockdownEvent` is useless. Furthermore, asking a trading AI to monitor AWS CloudTrail logs and edit VPC Security Groups is a massive violation of the Single Responsibility Principle. A trading bot should trade; it should not be a Cloud Security Posture Management (CSPM) tool.

### Critical User Journeys (CUJs)
1. **The Too-Late Lockdown:** An attacker dumps the `prompts` table. 15 minutes later, CloudTrail logs the event. The CrewAI system notices it and locks the database. The attacker already has the alpha.
2. **The Self-Denial of Service (DoS):** The LLM misinterprets a legitimate health-check ping from AWS Route53 as an "unauthorized IP address." It autonomously edits the VPC Security Group to block all inbound traffic. The entire trading infrastructure is instantly severed from the Alpaca API.

## 🛠️ Refinement & Architecture Update
To make this robust:
1. **Remove Cloud Security from AI Scope**: The CrewAI system is completely stripped of any ability to read AWS logs, modify IAM roles, or touch Security Groups. The AI must be isolated strictly to the application layer.
2. **Standardized Infrastructure Security**: Network security is handled by standard, deterministic AWS services: AWS WAF, VPC Endpoints, and GuardDuty. If GuardDuty detects an anomalous database query, it triggers an AWS Lambda function to block the IP. The AI is entirely unaware of the network layer, ensuring it remains hyper-focused on financial execution without risking catastrophic infrastructure self-sabotage.
